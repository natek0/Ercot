# Stage 5 — Engineering / TDD-CI/CD Review

**Reviewer lens:** staff engineer against a real software-engineering bar (an explicit role
requirement). Question asked: would this code pass review at a serious shop, and does the
reproducibility claim ("one command regenerates every number and figure") actually hold?

**How I verified:** ran `python -m pytest` (96 passed, 18 in Stage 5), `python -m src.stage5_run`
(off cache), `python -m src.figures` (6 PNGs), timed the cached report, timed the bootstrap,
and read all five Stage 5 source/test files plus the writeup, notes, `MPC_2H`, `reserves`,
`.gitignore`, and `requirements.txt`.

**Bottom line.** The inference math is correct and the numbers reproduce bit-for-bit off the
cache; the pure/heavy split is genuinely clean and the honest-reporting discipline is intact
(no manufactured significance). **No correctness blockers.** The gaps are engineering-quality
and reproducibility ones: a 30-second "instant" report from an un-vectorized pure-Python
bootstrap, a ladder figure whose comparator bars are hardcoded constants while the true values
are computed-and-thrown-away one function earlier, figures that are untracked (broken images on
a fresh clone), and a test suite that pins logic well but never checks the one property that
matters most — that the bootstrap CI is actually *calibrated*.

Ranking used (per brief): correctness bug > reproducibility gap > missing meaningful test >
quality/polish.

## Findings

### F1 — `fig_ladder` comparator bars are hardcoded constants; the true values are computed and discarded
- **What:** The value-of-foresight ladder (Fig 1) and the §1 writeup table's full-window rows
  (ceiling $13,206, clairvoyant MPC $12,847, learned −$303, naive −$3,453, floor −$5,267) are
  read from a hand-typed dict, not regenerated by the Stage 5 pipeline.
- **Where:** `src/figures.py:74` `from src.stage4_run import MPC_2H as M`; `src/figures.py:81-82`
  uses `M["ceiling"], M["clair"], M["learned"], M["naive"], M["floor"]`.
  `src/stage4_run.py:38` `MPC_2H = {"ceiling": 13206, "clair": 12847, "learned": -303, ...}`.
- **Why it's wrong:** `build_cache` **already runs** the full-window backtests that produce four
  of those five numbers — `r_ml`, `r_mn`, `r_fl` at `src/stage5_run.py:108-112` — and each
  `BacktestResult` exposes a full-window `.profit` (`src/backtest.py:27,90`; used elsewhere at
  `stage5_run.py:157`). But `build_cache` calls `_daily_pnl(r_ml.log, mask)` and **discards
  `r_ml.profit`**. So the pipeline computes the honest full-window learned/naive/floor totals,
  throws them away, and the figure re-imports frozen copies from a different module. The
  writeup's §8 promise — "reproduce every number and figure" with one command — is therefore
  literally false for the ladder's comparator bars and the §1 table: a `--rebuild` that shifted
  the data would leave those bars stale, silently. This is exactly the class of drift an
  interviewer probes ("what regenerates this number?").
- **Fix (correct because it closes the loop with data already in hand):** in `build_cache`,
  stash the full-window profits into the meta parquet next to the ceilings —
  `pd.DataFrame({..., "clair": [clair], "learned_full": [float(r_ml.profit)],
  "naive_full": [float(r_mn.profit)], "floor_full": [float(r_fl.profit)]})` — and have
  `fig_ladder` read them from `daily.attrs`/meta instead of `MPC_2H`. `ceiling` is already
  recomputed (`stage5_run.py:123`, equals 13,206). Only the clairvoyant-MPC number ($12,847) is
  a genuine external citation (no clairvoyant-MPC backtest is run in Stage 5) and should be
  labelled as a cited Stage-4 constant, not blended into a dict that looks recomputed. This
  deletes five magic numbers and a cross-module constant import.
- **Severity:** major (reproducibility gap). **Confidence:** verified. **Interview value:**
  raises-bar (removing "hidden constants that look computed" is the single most credible
  reproducibility fix here).

### F2 — Cached report takes ~30 s, not "instant"; pure-Python stationary bootstrap is the cost
- **What:** `python -m src.stage5_run` (no `--rebuild`, pure cache read) takes **30.5 s wall**.
  The notes (`reports/stage5_notes.md:35`) and the task brief both say the cached path "prints
  instantly."
- **Where:** `src/stage5_stats.py:86-97` — `stationary_bootstrap_samples` is a triple-nested
  Python loop (`for s in n_boot: while i<N: for k in L:`), ~700k Python-level index writes per
  call at N=140/n_boot=5000. Measured: 0.27 s per `bootstrap_ci`, **2.1 s per 6-block sweep**.
  `report()` fires two full block sweeps (`paired_report` for the edge at
  `stage5_run.py:213`/line 211 inside, and the kernel-gap `paired_report` at `:253`), plus
  level/mean/psi CIs — the sweeps dominate the 30 s.
- **Why it's wrong:** a "cached, instant" claim that is actually 30 s is a small honesty gap in a
  writeup that otherwise trades on precise honesty, and the loop is O(n_boot·N) interpreted work
  where it should be O(1) NumPy. At the fleet/summer extension the brief scopes (300 assets ×
  more days) this becomes minutes.
- **Fix (correct because the stationary bootstrap is a pure index construction):** vectorize the
  index generation. One clean form: draw all block starts and a geometric "new-block" mask for
  the whole `n_boot×N` grid at once, then `np.where(new_block, random_start,
  prev_index+1) % N` via a cumulative construction; or, simplest and fully equivalent, build the
  `n_boot×N` index matrix with `rng.integers` for starts + `rng.random < p` for block breaks and
  a `np.cumsum`-based within-block offset, then `stat_fn(x[idx], axis=1)`. Keep the same seed
  semantics so numbers stay reproducible (re-pin the golden values in a test after). Expected
  ~20–50× speedup → sub-second report.
- **Severity:** major (reproducibility-claim + scalability). **Confidence:** verified.
  **Interview value:** raises-bar (vectorizing a Monte-Carlo inner loop and *re-pinning the
  seeded output in a test* is precisely the "TDD + performance" signal the role names).

### F3 — Figures are untracked; the writeup embeds them → broken images on a fresh clone / GitHub
- **What:** `reports/figures/*.png` are the artifact an interviewer sees (the writeup embeds all
  six via `figures/figN_*.png`), but they are **untracked** in git (`git ls-files reports/figures/`
  is empty; `git status` shows `?? reports/figures/`) and are **not** gitignored.
- **Where:** `.gitignore:11` ignores `data/raw/` (the cache) — correct — but nothing ensures the
  figures are committed. Regeneration needs the cache, which needs the raw ERCOT panel
  (gitignored, re-ingest from API). So a fresh clone renders the writeup with six broken images
  until someone re-ingests + rebuilds (~8 min + API access).
- **Why it's wrong:** the writeup is the deliverable; a reader on GitHub sees broken image links.
  Either the figures are committed (they are the output, like a paper's PDF figures) or the
  writeup should not hard-link them. Current state is neither.
- **Fix:** `git add reports/figures/*.png` and commit them as deliverables (they are small,
  ~50 KB each, and regenerable — the standard "commit the rendered artifact, gitignore the
  intermediate cache" split). This is consistent with the existing choice to track the writeup
  and notes but gitignore `data/raw/`.
- **Severity:** major (reproducibility/hygiene). **Confidence:** verified. **Interview value:**
  raises-bar (a paper whose figures render on first view; the opposite reads as unfinished).

### F4 — Highest-value missing test: the bootstrap CI is never checked for *coverage*
- **What:** `test_bootstrap_ci_covers_mean_of_known_normal` (`tests/test_stage5.py:67-71`) is
  named "covers" but only asserts `lo > 0` for one loc=10 dataset — a *direction* check, not a
  *coverage* check. Nothing verifies the CI attains ~95% nominal coverage, which is the entire
  inferential claim the headline rests on ("the edge CI straddles zero" only means something if
  the interval is calibrated).
- **Where:** `tests/test_stage5.py:67-77`; the routine under test is `bootstrap_ci` /
  `stationary_bootstrap_samples` (`stage5_stats.py:70-113`).
- **Why it's wrong:** a mis-scaled or mis-seeded bootstrap would still pass every current test
  (they check determinism, bracketing, and one-sided direction) yet silently under- or
  over-cover, invalidating every CI in the paper. This is the classic "the test name promises a
  property the assertion doesn't enforce" trap.
- **Fix (a real property test):** Monte-Carlo coverage. Draw M≈300 datasets of n≈120 from a
  known DGP (e.g. i.i.d. Normal(μ,σ), and separately an AR(1) to exercise dependence), build the
  95% CI on the sum each time with a *different* seed, and assert the empirical coverage of the
  true sum-parameter is within a binomial tolerance band of 0.95 (e.g. [0.90, 0.985] at M=300).
  Keep it fast (n_boot≈500). This pins the one property the study's credibility depends on.
- **Severity:** major (missing meaningful test on the load-bearing routine). **Confidence:**
  verified (read the assertion). **Interview value:** raises-bar (a calibrated-coverage test is
  the difference between "I used a bootstrap" and "I validated my bootstrap").

### F5 — The IO plumbing that builds the headline series (`_traded_mask`, `_daily_pnl`) is untested
- **What:** Every headline number flows through `_traded_mask` (drops the 2-month warm-up) and
  `_daily_pnl` (masks + groups interval P&L to calendar-day). Neither has a unit test. The notes
  claim "no future leaks" and "matched window" as the Stage-4 key fix, but that fix lives
  entirely in these two untested functions.
- **Where:** `src/stage5_run.py:59-73`. Only the pure `stage5_stats` layer is tested; the brief
  explicitly asks whether the mask/alignment is guarded — it is not.
- **Why it's wrong:** an off-by-one in `months[min_train_months]` (e.g. `>=` vs `>`, or a
  timezone slip in `to_period("M")`) would silently change which days are "traded," moving the
  headline, with zero test to catch it. And `_daily_pnl` assumes `len(log) == len(mask)` and
  positional alignment between the backtest log and the panel — an implicit contract with no
  assertion.
- **Fix:** a synthetic test: build a tiny fake `panel` spanning 4 months (15-min `ts`, `date`
  column) and a matching fake `log` with a known `step_profit` pattern; assert (a)
  `_traded_mask` drops exactly the first two months and keeps the rest, (b) `_daily_pnl` returns
  one row per traded calendar day with the correct summed profit, (c) a deliberately
  short/reordered log is caught. Add `assert len(log) == mask.size` at the top of `_daily_pnl`
  as the cheap runtime guard.
- **Severity:** major (missing test on the leak-free/matched-window contract — the thing the
  Stage-4 review specifically fixed). **Confidence:** verified. **Interview value:** raises-bar
  (testing the alignment/leak boundary, not just the stats kernel, is what "leak-free
  walk-forward, tested" actually means).

### F6 — Dead `med_ci`; docstring promises a "day-clustered median CI" that is neither reported nor day-clustered
- **What:** `report_psi` computes a 5,000-replicate bootstrap CI on the median and never prints
  or returns it; and its docstring says it reports "(i) a **day-clustered** bootstrap CI on the
  **MEDIAN** of the binding intervals," but the median CI (a) is dead and (b) uses the plain
  interval-level stationary bootstrap on `np.where(psi_iv>tol, psi_iv, 0)` — not day-clustered,
  and taken over an array that is ~94% zeros so its median is 0 anyway.
- **Where:** `src/stage5_run.py:286` (`med_ci = st.bootstrap_ci(...)`, never used);
  docstring `stage5_run.py:269-274`.
- **Why it's wrong:** dead code that costs a real 5,000-resample bootstrap on every report run,
  plus a docstring that describes output the function does not produce — both are review-blockers
  at a serious shop (they mislead the next reader about what ψ_up inference was actually done).
- **Fix:** either (a) delete `med_ci` and fix the docstring to describe what is actually reported
  (median over binding intervals as a point stat; a bootstrap CI on the *mean daily-max*, which
  IS day-level and IS printed), or (b) if a median CI is wanted, compute it on `pos` (the binding
  intervals only) and print it, and make the day-clustering real by bootstrapping over *days*
  (block over the daily-max series) rather than intervals. (a) is the minimal honest fix.
- **Severity:** minor (dead code + doc/behavior mismatch; no wrong number is *printed*).
  **Confidence:** verified. **Interview value:** neutral (cleanliness; but a reviewer will spot
  it and it undercuts the "carefully honest" tone).

### F7 — Test/count drift in the writeup and a near-tautological test
- **What:** (a) `reports/stage5_writeup.md:251` says "**95 tests** incl. 17"; the suite is
  **96** (`pytest --collect-only`) with **18** in `tests/test_stage5.py`. The notes
  (`stage5_notes.md:175`) also say "95 passed / 17 Stage 5." (b)
  `test_block_length_sensitivity_shape` only asserts the `block_mean` values are echoed back in
  order — it pins nothing about the CIs.
- **Where:** `reports/stage5_writeup.md:251`, `reports/stage5_notes.md:175`;
  `tests/test_stage5.py:80-83`.
- **Why it's wrong:** a wrong test count in the reproducibility section is a small credibility
  ding in a doc whose whole pitch is "every number is checked." The shape test is close to
  tautological (it can't fail unless the list comprehension is deleted).
- **Fix:** update the counts to 96/18. Strengthen the shape test into a *dependence* test: on an
  AR(1) series the 20-day-block CI width should exceed the 1-day (i.i.d.) width — which is the
  entire justification for using the stationary bootstrap over an i.i.d. one, and currently
  nothing asserts it.
- **Severity:** minor. **Confidence:** verified. **Interview value:** neutral (the AR(1)
  width-monotonicity test is the one worth adding — it demonstrates you know *why* the stationary
  bootstrap is the right tool).

### F8 — Headless robustness: matplotlib backend not forced; minor bootstrap edge case
- **What:** (a) `src/figures.py` never sets a non-interactive backend in `main()`/`_style()`
  (only the *test* forces Agg, `tests/test_stage5.py:172`). On a machine that selects a GUI
  backend this can warn/stall; in CI-without-display it happens to work because matplotlib
  auto-picks Agg, but that's luck, not a contract. (b) `stationary_bootstrap_samples` returns
  `np.array([])` for N==0 (`stage5_stats.py:81-82`), then `bootstrap_ci` calls
  `np.percentile([], ...)` → NaN + warning. Not reachable with real data, but it's an unguarded
  edge.
- **Where:** `src/figures.py:44-55,223-234`; `src/stage5_stats.py:81-82,108-109`.
- **Why it's wrong:** the reproducibility claim includes "on a fresh machine / in CI"; forcing
  Agg makes figure generation deterministic and headless-safe by contract rather than by
  accident.
- **Fix:** `import matplotlib as mpl; mpl.use("Agg")` at the top of `figures._style()` (or
  `main()`); guard `bootstrap_ci` to return NaN dict when `sims.size == 0`.
- **Severity:** minor/polish. **Confidence:** verified. **Interview value:** cosmetic (but the
  Agg line is a one-liner that signals "I've shipped headless plotting before").

### F9 — Magic numbers scattered, no module-level provenance
- **What:** ρ=0.05 appears as a literal in three places (`stage5_run.py:91,106,142`), the ψ_up
  tolerance `0.01 * median_price` and the `32.75` Stage-0 comparator are inline literals
  (`stage5_run.py:277,296`; `figures.py:211`), and `N_S`, `N_BINS` are module constants but the

- **Where:** as above.
- **Why it's wrong:** these are pre-registered/decision-of-record values (ρ sweep default; the
  Decision-4 1%-of-median tolerance; the Stage-0 max). A reader can't tell which literals are
  free parameters vs frozen decisions. Not a bug — the values are correct and match the notes.
- **Fix:** promote to named module constants with a one-line comment citing the decision
  (`RHO_DEFAULT = 0.05  # Decision-17 sweep default`,
  `STAGE0_PSI_MAX = 32.75  # reports/step0_results.md`,
  `PSI_TOL_FRAC = 0.01  # Decision 4`). Cheap, and it documents provenance where it's used.
- **Severity:** polish. **Confidence:** verified. **Interview value:** cosmetic.

## Things Stage 5 gets RIGHT (stated plainly, not invented work)

- **The pure/heavy split is genuinely clean.** `stage5_stats.py` is numpy-in/numbers-out with no
  I/O, so the inference logic is unit-testable in isolation; the heavy leak-free backtests are
  quarantined in `stage5_run.py`. This is the right architecture and it's executed properly.
- **Determinism holds.** Every randomized routine takes `seed=0`; `test_bootstrap_is_seeded_
  deterministic` pins byte-identical dicts; I re-ran the report and got identical numbers to the
  writeup. Reusing `seed=0` across the block-length sweep is a *feature* (it holds Monte-Carlo
  noise fixed so the sweep isolates the block-length effect), not a bug.
- **Honest reporting is intact.** The CI straddles zero, the sign test is a coin flip, and the
  code reports all three without a single knob that would manufacture significance. The
  concentration function deliberately reports both `share_of_total` and `share_of_gross` *because*
  the net total can be small — that's a thoughtful guard against the exact way tail-concentrated
  P&L misleads. No §VIII.5 discipline violation found.
- **The cache workaround is sound.** parquet `attrs` don't round-trip, so ceilings are stashed in
  a sidecar `_meta.parquet` and ψ_up in an `.npz` — a legitimate, documented pattern, and
  `_load_cache` restores them correctly.
- **The figures smoke test is a real guard,** not a rubber stamp: it drives every figure function
  through synthetic data on the Agg backend and asserts non-empty PNGs, which is what caught the
  mathtext/currency-escaping bugs the docstring references. It just doesn't *replace* the missing
  coverage/alignment tests above.
- **No leakage found in the Stage-5-specific code path.** The mask/warm-up logic is correct; the
  discipline is real. My F5 is that it's *untested*, not that it's *wrong*.

## Resume / interview-value upgrades (engineering lens)

Ranked by signal-per-line to a Base Power Markets interviewer.

1. **Vectorize the stationary bootstrap and re-pin its seeded output in a test (F2 + F4).** This
   is the single highest-value upgrade: it turns a 30 s interpreted loop into a sub-second NumPy
   kernel *and* the accompanying coverage/golden test demonstrates you validate Monte-Carlo code
   the way you'd validate a pricing library. Optimization + statistics + TDD in one change — the
   exact triad the role names. **Signal: performance-aware numerical engineering with a
   correctness proof.**
2. **Make the ladder fully data-driven (F1) and commit the figures (F3).** Together these make
   "one command regenerates every number and figure" *true*, which is the claim the writeup
   stakes its credibility on. An interviewer who runs the repo should get the paper's exact
   numbers with no hidden constants. **Signal: reproducibility you can actually demonstrate
   live.**
3. **Add the alignment/leak-boundary test (F5).** Testing `_traded_mask`/`_daily_pnl` — the
   matched-window fix the Stage-4 review specifically demanded — shows you test the *contract that
   matters* (no lookahead, correct day alignment), not just the easy pure functions. **Signal:
   you understand that the leak-free claim is only as good as the test guarding it.**
4. **Wire a fast synthetic Stage-5 smoke into CI.** CI currently exercises only the pure stats +
   the figures smoke; the heavy path (`build_cache` → `report`) has zero CI coverage because it
   needs the real panel. Add a `build_cache(synthetic_panel)` path (tiny random price panel, a
   few weeks) that runs the *whole* orchestration end-to-end in a few seconds, asserting the
   report dict has the right keys and finite numbers. **Signal: CI that guards the integration
   seam, not just the units — "CI/CD" as the posting words it.**
5. **A one-line AR(1) width-monotonicity test for the block bootstrap (F7).** Cheap, and it
   demonstrates you can articulate *why* the stationary bootstrap beats an i.i.d. one on
   serially-dependent P&L — a question a quant interviewer will actually ask.
